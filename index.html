How I build with Claude Code

01

Start with the problem, not the code

I describe the business problem and success criteria before writing a single line. The quality of the input determines the quality of the output — so I spend real time on requirements before prompting.

02

Architecture first, implementation second

I have Claude Code propose the system architecture — data models, API boundaries, component structure — and review it against what I know about performance, scalability, and clean data flow before any implementation.

03

Review everything the AI generates

The most important skill. I read every file Claude Code produces. I catch unindexed queries, bloated responses, missing error handling, and poorly structured components. I know what good code looks like — and I redirect the AI when it misses.

04

Debug from logs and stack traces

When something breaks, I read the error, trace it through the system, understand the root cause, and direct Claude Code to the targeted fix. I don't blindly re-prompt — I diagnose first.

05

Ship it, then iterate

Every project below is deployed and running. I push to production, monitor for issues, and iterate. The goal is working software, not mockups.

Projects

Alpaca RAG Trader

Autonomous paper trading agent combining retrieval-augmented generation with live market data — built entirely with Claude Code.

Python SQLite FTS5 Claude API Alpaca API Chart.js stdlib only
Market Data (Alpaca) ──┐ Portfolio State ──┼──▶ Reasoner (Claude Sonnet) ──▶ Risk Validator ──▶ Broker (Alpaca) Knowledge Base (FTS5) ──┘ ▲ │ 6,906 chunks from 4 investing books

What it does

  • Ingests 4 investing books (6,906 text chunks) into a SQLite FTS5 knowledge base with BM25 ranking
  • Each cycle: fetches live market data, gets portfolio state, queries relevant book passages, sends context to Claude Sonnet for structured JSON trade decisions with confidence scores
  • Enforces risk guardrails — max 10% per position, $5,000 max per trade, 0.6 minimum confidence threshold
  • Live monitoring dashboard with portfolio metrics, positions, trade history, and confidence distributions

How I built it with Claude Code

  • Described the full system first — "autonomous trading agent that consults a book knowledge base before making decisions" — then iterated on architecture until the data flow was clean
  • Deliberately avoided heavy C-extension packages (chromadb, numpy, pydantic). Directed Claude Code to use stdlib: SQLite FTS5 over vector databases, dataclasses over Pydantic, urllib over httpx
  • Caught and corrected AI-generated issues: deprecated Alpaca endpoints, mid-sentence chunking, and loose risk validation logic
  • Built the monitoring dashboard last — described needed metrics, reviewed Chart.js code, iterated on auto-polling for market hours

Key judgment calls

  • Chose FTS5 over vector embeddings — simpler, zero dependencies, and BM25 is good enough for structured book content. The kind of call you make when you understand AI tool strengths vs. actual requirements
  • Added dry-run mode early to validate the full pipeline without real orders — a systems thinking decision that saved debugging time later

Cashflow Tracker

Personal finance app with interactive dashboards for transaction tracking, categorization, and financial reporting.

React TypeScript Vite

What it does

  • Interactive dashboards for tracking transactions, assigning categories, and generating financial reports
  • Type-safe with React + TypeScript, fast iteration with Vite
  • Clean component architecture with reusable UI patterns

How I built it with Claude Code

  • Scaffolded the full React + TypeScript + Vite project with Claude Code, then reviewed and restructured the component hierarchy for maintainability
  • Developed my debugging workflow here: read the error, understand why (usually a TypeScript type mismatch or missing prop), then give Claude Code a targeted fix instead of a vague "fix this"
  • Learned prompt specificity matters — "add a bar chart" produces worse code than "add a bar chart component that takes monthly expense data grouped by category, renders with accessible colors, and handles empty states"

Moltbook Daily Scraper

Automated agent that scrapes Moltbook daily, uses Claude AI to identify promising ideas, and emails a curated summary — running on AWS Lambda.

Python BeautifulSoup Claude API AWS Lambda CloudWatch SMTP
CloudWatch (daily cron) ──▶ Lambda ──▶ Scraper (BeautifulSoup) ──▶ Analyzer (Claude API) ──▶ Emailer (SMTP → inbox)

What it does

  • Scrapes Moltbook posts with BeautifulSoup, evaluates each for innovation, financial potential, feasibility, and risks via the Claude API
  • Delivers curated HTML email digests with the top 5 ideas ranked and analyzed
  • Runs on AWS Lambda + CloudWatch Events, daily at 12:00 UTC — total cost ~$1.50/month
  • Modular architecture: scraper.py, analyzer.py, emailer.py — each independently testable with fallback logic

How I built it with Claude Code

  • Described the end-to-end pipeline and had Claude Code generate the modular architecture. Reviewed separation of concerns and ensured each module was independently testable
  • Lambda deployment needed the most correction — IAM policy too permissive, cold start handling wrong. Read CloudWatch logs, identified issues, directed targeted fixes
  • Added configurable parameters (MAX_POSTS, TOP_N_IDEAS, SMTP settings) as a design decision I made before prompting — not something the AI suggested
